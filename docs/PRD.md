# 赛博忏悔室 (Cyber Confessional) V2.0 产品需求文档 (PRD)

## 核心产品愿景
将冷酷的结构因果模型 (SCM) 与宗教的告解仪式感相结合，通过多维度的功能矩阵，帮助用户完成从“被动接受暴击”到“主动推演未来”，再到“获得群体共鸣”的心智升级。

---

## 阶段一：命运推演沙盘 (The Intervention Sandbox)
**难度级别：⭐（低难度，见效极快，强烈推荐首发）**

### 1. 核心概念
打破单向的“宿命宣判”，赋予用户上帝视角。让用户亲手拨动环境 ($Z$) 和传导行为 ($M$) 的滑块，直观感受“反事实概率”的剧烈跳动，从而真正理解“改变结构胜过改变选择”的硬核道理。

### 2. 功能详情
*   **前端 UI (控制台面板)：** 在右侧的参数解析卡片下方，新增两个赛博朋克风格的滑块（Slider）。
    *   **环境干预器 ($Z$)：** 默认在当前 $Z$ 值，两端标注“顺风局”和“绝境”。
    *   **行为阻断器 ($M$)：** 默认在当前 $M$ 值，两端标注“绝对理性”和“致命失控”。
*   **交互逻辑：** 当用户拖动滑块时，右下角的 Mermaid 因果图的对应连线会变粗/变细（或改变发光强度），同时大字体的“干预成功率”实时跳动。

### 3. 技术实现路径
*   **前端：** 引入 React Slider 组件，监听 `onChange` 事件。
*   **后端：** 新增一个极轻量级的 API `/api/simulate`。这个接口**完全不需要调用 LLM**，只需接收前端传来的新 $Z$、$M$ 参数和原有的结构方程权重，直接实例化 `ComplexCausalEngine` 运行一次 `run_inference` 并返回概率。毫秒级响应，体验丝滑。

## 阶段二：叹息之壁 (Wall of Lamentations)
**难度级别：⭐⭐（中低难度，需引入轻量级数据库）**

### 1. 核心概念
打造一个匿名、去中心化的“遗憾展览馆”。人类在面对被数学证明的失败时，最大的心理安慰往往来自于“看到别人也被同样碾压”。这不仅能提供情绪价值，更是绝佳的社交裂变点（Shareable Content）。

### 2. 功能详情
*   **信息流广场 (The Feed)：** 主页新增一个入口通向“叹息之壁”。这里以瀑布流或弹幕的形式展示历史信徒的告解。
*   **展示卡片内容：** 
    *   去隐私化的一句话告解（如：“当初没买石油”）。
    *   核心暴击数据（如：反事实成功率 0.02%，隐变量：极度风险厌恶 2.84）。
    *   一张缩略版的赛博因果图 (DAG)。
*   **共鸣交互：** 卡片底部有类似“点赞”的按钮，名为 `[Amen / 确证收束]`。

### 3. 技术实现路径
*   **后端 & 数据库：** 引入轻量级的数据库（如 SQLite 或 PostgreSQL）。建一张 `confessions` 表，字段包括 `text`, `u_value`, `z_name`, `prob`, `mermaid_str`, `amen_count`。
*   **数据流转：** 用户在专属告解室看完判词后，点击“将遗憾刻入叹息之壁”，前端向后端发送 POST 请求落库；广场页通过 GET 请求拉取分页数据。

## 阶段三：诘问模式 (Socratic Inquisitor)
**难度级别：⭐⭐⭐（中高难度，涉及 LLM 状态机与多轮会话）**

### 1. 核心概念
改变目前“一波流”的输入输出方式。如果用户提供的信息太短（例如只说“我后悔分手了”），大模型无法准确推导出 $Z$ 和 $M$，神父将化身为苏格拉底，通过追问补全因果拼图。

### 2. 功能详情
*   **对话式告解：** 核心交互区从“大输入框”变为“多轮对话流”。
*   **智能追问：** 神父的回复不再直接是判词，而是像剥洋葱一样探寻细节：“孩子，你只说了结果，但那天你们所处的环境 ($Z$) 是怎样的？在说出那句话之前，你的情绪传导 ($M$) 经历了什么？”
*   **收束宣判：** 当收集到足够的信息后，神父才会触发后端的 `ComplexCausalEngine`，屏幕瞬间切换至红蓝交织的“审判状态”和数据面板。

### 3. 技术实现路径
*   **LLM 状态机设计：** 需要修改 Prompt，让 LLM 输出带有状态码的 JSON。例如：`{"status": "NEEDS_INFO", "question": "追问内容"}` 或 `{"status": "READY", "graph_params": {...}}`。
*   **前端状态管理：** 前端需要维护一个 `messages` 数组来渲染聊天界面，并根据后端返回的 `status` 决定是继续显示聊天框，还是展示右侧的赛博参数图表。

## 阶段四：赛博告解录 (The Book of Sins)
**难度级别：⭐⭐⭐⭐（高难度，完整的全栈应用与长文本分析）**

### 1. 核心概念
从“单次算命”跃升为“人生的量化体检”。通过建立用户账户系统，神父能够追踪同一个用户在不同时间、不同事件下的隐变量 ($U$)，从而给出终极的性格宿命论断。

### 2. 功能详情
*   **灵魂档案 (User Profile)：** 用户可以注册/登录。拥有自己的专属档案页，查阅过去所有的告解记录和因果图。
*   **隐变量均值追踪：** 面板上展示用户历史 $U$ 值的折线图或雷达图。
*   **阶段性审判报告：** 当用户完成特定数量（如 5 次）的告解后，生成一份深度报告：“在职业、感情和投资的三次分支中，表象各异，但你的底层隐变量 [风险极度厌恶] 均值高达 3.5。这是你逃不开的引力场。”

### 3. 技术实现路径
*   **鉴权系统 (Auth)：** 引入 JWT Token 或成熟的鉴权服务（如 Supabase Auth / Clerk / NextAuth），实现注册、登录、Session 管理。
*   **数据库重构：** 建立关联型数据库结构：`Users` (1) -> (N) `Confessions`。
*   **高级 LLM 总结能力：** 增加一个新的专属 Prompt `PROMPT_HISTORICAL_ANALYSIS`。当触发体检报告时，后端将该用户过去 5 次的 `[事件 X, 结局 Y, 隐变量 U]` 打包发给 LLM，让其寻找跨越不同事件的深层共性。

### 🚀 迭代建议 (Next Steps)

按照产品开发的 MVP 原则，我强烈建议：
1.  **本周：** 先把**阶段一（推演沙盘）**做了。只需要加两个滑块和一个简单的算力接口，就能立刻让产品的可玩性翻倍，给用户极大的震撼。
2.  **下周：** 顺手加上**阶段二（叹息之壁）**。不需要做登录，直接匿名上墙，方便你做产品发布初期的社交媒体引流和裂变。
3.  **未来：** 等产品有了几千个自然流量的信徒，再慢慢去啃**阶段三**和**阶段四**这种需要重构状态机和用户账户系统的硬骨头。
